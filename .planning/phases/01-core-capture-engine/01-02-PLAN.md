  <name>Task 1: Browser lifecycle, auto-scroll, and output modules</name>
  <files>src/browser.ts, src/scroll.ts, src/output.ts, tests/browser.test.ts, tests/scroll.test.ts, tests/output.test.ts</files>
  <behavior>
    Browser (tests/browser.test.ts):
    - getBrowser() returns a connected Browser instance
    - getBrowser() called twice returns the same instance (singleton)
    - closeBrowser() disconnects the browser
    - After closeBrowser(), next getBrowser() launches a new browser
    - registerCleanup() registers SIGINT/SIGTERM handlers (verify handlers are attached)

    Scroll (tests/scroll.test.ts):
    - autoScroll(page) scrolls to bottom then back to top (verify scrollY === 0 after)
    - autoScroll on lazy-page.html triggers image loading (images have naturalWidth > 0)
    - autoScroll completes within timeout (does not hang)

    Output (tests/output.test.ts):
    - generateFilename("https://example.com/page", "png") returns "example-com_{timestamp}.png"
    - generateFilename with jpeg format returns .jpg extension
    - resolveOutputPath with custom dir and filename uses them directly
    - resolveOutputPath with no dir defaults to cwd
    - resolveOutputPath with no filename generates one
    - ensureOutputDir creates directory recursively if missing
    - File collision: writing to existing path overwrites silently
  </behavior>
  <action>
1. Implement `src/browser.ts`:
   - Module-level `let browser: Browser | null = null`
   - `getBrowser()`: lazy-launch Chromium (headless: true), return singleton. If browser exists but `!browser.isConnected()`, re-launch.
   - `closeBrowser()`: close browser if connected, set to null
   - `registerCleanup()`: attach SIGINT/SIGTERM handlers that call closeBrowser() then process.exit(0). Also attach `uncaughtException` handler that kills browser PID as a last resort (per research recommendation).
   - All logging to stderr (NEVER stdout -- MCP stdio transport uses stdout)

2. Implement `src/scroll.ts`:
   - `autoScroll(page: Page): Promise<void>` -- runs page.evaluate() with incremental scrolling
   - Scroll step: 250px, delay: 100ms between steps
   - After reaching bottom, scroll back to top (window.scrollTo(0, 0))
   - After scrolling, call `page.waitForLoadState("networkidle")` with a catch (timeout is OK)
   - Total scroll timeout: 10 seconds max (prevent infinite scroll pages from hanging)

3. Implement `src/output.ts`:
   - `generateFilename(url: string, format: "png" | "jpeg"): string` -- parse URL hostname, replace dots with hyphens, add ISO timestamp (formatted as YYYY-MM-DD_HH-mm-ss), add extension (.png or .jpg)
   - `resolveOutputPath(outputDir?: string, filename?: string, url: string, format: "png" | "jpeg"): string` -- resolve dir (default cwd), resolve name (default generated), return path.resolve(dir, name)
   - `ensureOutputDir(dirPath: string): Promise<void>` -- fs.mkdir with recursive: true

4. Replace test stubs in browser.test.ts, scroll.test.ts, output.test.ts with real tests matching the behavior spec above. Use the HTML fixtures from Plan 01 for scroll tests (file:// URLs).

5. Run all three test files and fix until green.

<task type="auto" tdd="true">
  <name>Task 2: Core capture pipeline</name>
  <files>src/capture.ts, tests/capture.test.ts</files>
  <behavior>
    - captureScreenshot({url: "file://test-page.html", fullPage: false, format: "png", ...defaults}) returns CaptureResult with valid PNG buffer
    - captureScreenshot with fullPage: true captures full scrollable height (buffer size > viewport-only)
    - captureScreenshot with format: "jpeg" produces JPEG buffer (starts with FF D8)
    - captureScreenshot with format: "jpeg", quality: 10 produces smaller file than quality: 90
    - captureScreenshot with width: 800, height: 600 produces screenshot at those dimensions
    - captureScreenshot with scale: 2 produces image with 2x pixel dimensions
    - captureScreenshot with outputDir and filename saves file at that path
    - captureScreenshot with wait: "domcontentloaded" does not wait for networkidle
    - captureScreenshot with wait: 500 (number) waits 500ms after load
    - captureScreenshot with fullPage: true, autoScroll: true triggers lazy content
    - captureScreenshot with fullPage: true, autoScroll: false skips scrolling
    - captureScreenshot on timeout captures whatever is loaded and sets warning field
    - captureScreenshot result includes filePath pointing to a file that exists on disk
    - captureScreenshot result includes base64-encodable buffer
  </behavior>
  <action>
1. Implement `src/capture.ts`:
   - Export `captureScreenshot(params: CaptureParams): Promise<CaptureResult>`
   - Get browser via `getBrowser()` from browser.ts
   - Create new browser context with viewport {width, height} and deviceScaleFactor from params
   - Create new page in context
   - Navigate with page.goto(url). For wait strategy:
     - If string ("networkidle", "domcontentloaded", "load"): use as waitUntil
     - If number: use waitUntil "load", then page.waitForTimeout(wait)
   - Wrap navigation in try/catch with timeout. On timeout: set warning "Page load timed out after {timeout}ms, capturing current state" -- do NOT throw (per user decision)
   - If fullPage && autoScroll: call autoScroll(page)
   - Take screenshot: page.screenshot({ fullPage, type: format, quality: format === "jpeg" ? quality : undefined })
   - Resolve output path via resolveOutputPath()
   - Ensure output directory exists via ensureOutputDir()
   - Write buffer to file with fs.writeFile
   - Get screenshot dimensions from the page viewport (or from the image metadata)
   - Close browser context in finally block
   - Return CaptureResult with filePath, buffer, format, width, height, warning

   - Clean up temp files after each test
